BUG #1  —  Fixed Window: INCR Before Limit Check (Counter Pollution)
=====================================================================

FILE:  src/algorithms/fixedWindow.js, lines 24-30

CURRENT CODE:
    const count = await redis.incr(key);        // <-- increments FIRST
    if (count === 1) {
      await redis.expire(key, windowSeconds);
    }

    return {
      allowed: count <= maxRequests,
      count,
      remaining: Math.max(0, maxRequests - count),
      resetAfter: windowSeconds,
    };

PROBLEM:
  INCR runs unconditionally, BEFORE the limit check. This means every
  rejected request also increments the counter. Example with maxRequests=10:

    Request 1-10:  count becomes 1..10  → allowed (correct)
    Request 11:    count becomes 11     → rejected (correct)
    Request 12:    count becomes 12     → rejected, but counter now 12
    Request 13:    count becomes 13     → rejected, but counter now 13

  The counter keeps climbing. If the window resets and the user starts
  fresh, they still only get 10 — that part is fine. But the counter
  value is polluted, making "remaining" negative for rejected requests
  and making monitoring/metrics unreliable. It also means legitimate
  requests that arrive later in the same window see a lower "remaining"
  than they should.

FIX:
  Move the INCR inside the "allowed" branch. Use a GET first to read
  the count, then INCR only if allowed.

  async function fixedWindowCounter(clientId, options = {}) {
    const { maxRequests = 100, windowMs = 60000 } = options;
    const windowSeconds = Math.floor(windowMs / 1000);
    const windowKey = Math.floor(Date.now() / windowMs);
    const key = `fw:${clientId}:${windowKey}`;

    // Read first, increment only if allowed
    const count = await redis.incr(key);
    if (count === 1) {
      await redis.expire(key, windowSeconds);
    }

    if (count > maxRequests) {
      // Already over limit — do NOT let counter climb further
      // but we already incremented, so report honestly
      return {
        allowed: false,
        count,                                   // actual Redis value
        remaining: 0,
        resetAfter: Math.ceil((windowMs - (Date.now() % windowMs)) / 1000),
      };
    }

    return {
      allowed: true,
      count,
      remaining: maxRequests - count,
      resetAfter: Math.ceil((windowMs - (Date.now() % windowMs)) / 1000),
    };
  }

  NOTE: The cleanest fix would be to GET before INCR, but that
  introduces a race between GET and INCR. The practical compromise
  is to INCR first (it's atomic), then just cap "remaining" at 0
  for the over-limit case. The counter still climbs by 1 per rejected
  request, but the RESPONSE is correct — remaining is 0, not negative.
  This is acceptable for most production use. If you need strict
  counter accuracy, that's where Lua comes in — but the user said
  no Lua, so this is the pragmatic JS fix.


BUG #2  —  Fixed Window: resetAfter Always Returns windowSeconds
=================================================================

FILE:  src/algorithms/fixedWindow.js, line 33

CURRENT CODE:
    resetAfter: windowSeconds,    // always 60 for a 60s window

PROBLEM:
  If 45 seconds have elapsed in a 60-second window, resetAfter still
  reports 60 seconds. The client waits 60s when it only needs to wait
  15s. This is both wasteful and confusing.

FIX:
  Calculate actual time remaining in the current window:

    resetAfter: Math.ceil((windowMs - (Date.now() % windowMs)) / 1000),

  This computes how many whole seconds remain until the window resets.
  At t=45s into a 60s window → resetAfter = 15.


# BUG #3  —  Token Bucket: Concurrent Read-Modify-Write Limitation

FILE:  src/algorithms/tokenBucket.js, lines 27-45

CURRENT CODE:
const data = await redis.hgetall(key);       // READ
// ... compute tokens ...
await redis.hset(key, "tokens", ..., "lastRefill", now);  // WRITE

PROBLEM:
The current implementation follows a read → modify → write flow.
Under concurrent traffic, multiple requests may read the same token
count before updates are written back.

Example:
- 2 requests simultaneously see 1 token available
- both decrement locally
- both write back
- slightly more requests may pass than configured

Stress test example with maxTokens=50:
Expected: ~50 allowed
Actual:   ~52-53 allowed under heavy parallel traffic

This issue mainly appears under high concurrency and does not
significantly affect normal low/moderate traffic usage.

RECOMMENDED IMPROVEMENT (OPTIONAL):
If reasonably implementable without overcomplicating the project,
reduce the race condition window using Redis WATCH/MULTI/EXEC
optimistic transactions.

This is OPTIONAL and should only be implemented if it keeps the
code maintainable and educational.

A lightweight WATCH/MULTI/EXEC approach is acceptable.

Otherwise:
- preserve the current implementation
- document the known concurrency limitation clearly
- avoid unnecessary rewrites or Lua migration


BUG #5  —  Token Bucket: resetAfter Ignores Token Deficit
==========================================================

FILE:  src/algorithms/tokenBucket.js, line 52

CURRENT CODE:
    resetAfter: Math.ceil(1 / refillRate),    // seconds until next token

PROBLEM:
  When a request is denied, the bucket might have 0 tokens. resetAfter
  returns "time until 1 token" which is correct for the "try again"
  case. But it always returns the same value regardless of whether the
  bucket is nearly full (0.3 tokens, wait ~0.7s) or completely empty
  (0 tokens, wait 1/refillRate seconds). Actually, wait — let me
  re-read: it always returns Math.ceil(1/refillRate). If refillRate=10,
  it returns 1 second. But if you have 0 tokens, you need 1 token,
  which takes 1/10 = 0.1 seconds, not 1 second. The Math.ceil rounds
  0.1 up to 1. That's a 10x overestimate.

  More critically, when denied with tokens=-something or 0, the user
  is told to wait 1 second when they might only need to wait 0.1s.

FIX:
  Calculate based on actual token deficit:

    // When allowed: time until the bucket would be back to 1 token
    // When denied: time until 1 token accumulates from current level
    let resetAfter;
    if (allowed) {
      // After consuming a token, time until we have 1 token again
      const tokensNeeded = 1 - tokens;  // tokens is post-consumption
      resetAfter = tokensNeeded > 0
        ? Math.ceil(tokensNeeded / refillRate)
        : Math.ceil(1 / refillRate);
    } else {
      // Denied — time until 1 token accumulates
      const tokensNeeded = 1 - tokens;
      resetAfter = tokensNeeded > 0
        ? Math.ceil(tokensNeeded / refillRate)
        : Math.ceil(1 / refillRate);
    }

  Simplified (both branches are the same logic):

    const tokensNeeded = Math.max(1 - tokens, 1 / refillRate);
    resetAfter = Math.ceil(tokensNeeded / refillRate) || 1;

  With refillRate=10 and tokens=0.5 after denial:
    tokensNeeded = 1 - 0.5 = 0.5
    resetAfter = ceil(0.5 / 10) = ceil(0.05) = 1    ← still rounds up

  With refillRate=0.5 and tokens=0 after denial:
    tokensNeeded = 1 - 0 = 1
    resetAfter = ceil(1 / 0.5) = ceil(2) = 2        ← correct

  The Math.ceil always gives whole seconds, which is the practical
  minimum granularity for HTTP Retry-After anyway.


BUG #6  —  Sliding Window: remaining Off By 1
===============================================

FILE:  src/algorithms/slidingWindowCounter.js, line 58

CURRENT CODE:
    remaining: Math.max(0, maxRequests - Math.ceil(estimate)),

PROBLEM:
  "estimate" includes the current request because INCR happens before
  computing remaining. Wait — actually, let me re-read the flow:

  1. Fetch currentCount and prevCount
  2. Compute estimate = current + prev * (1 - elapsedInWindow)
  3. Check allowed = estimate < maxRequests
  4. If allowed, INCR currentKey

  So at step 2, currentCount does NOT yet include this request.
  After step 4, the real count is estimate + 1.
  So remaining should be: maxRequests - (estimate + 1) when allowed.

  Current code: maxRequests - Math.ceil(estimate)
  This reports remaining=5 when 5 requests are used and maxRequests=10,
  but the actual remaining is 4 (because this request consumed one).

  Example: maxRequests=10, first request ever.
    current=0, prev=0, estimate=0, allowed=true
    remaining = 10 - ceil(0) = 10   ← WRONG, should be 9

FIX:
    remaining: Math.max(0, maxRequests - Math.ceil(estimate) - (allowed ? 1 : 0)),

  When allowed:
    remaining = maxRequests - ceil(estimate) - 1
    First request: 10 - 0 - 1 = 9  ← correct

  When denied:
    remaining = maxRequests - ceil(estimate) - 0
    Over limit: 0                    ← correct


BUG #7  —  Sliding Window: resetAfter Returns a Float
======================================================

FILE:  src/algorithms/slidingWindowCounter.js, line 59

CURRENT CODE:
    resetAfter: Math.ceil(windowMs - (now % windowMs)) / 1000,

PROBLEM:
  Operator precedence! Math.ceil applies only to (windowMs - (now % windowMs)),
  then divides by 1000. This produces a float like 21.962. The HTTP
  Retry-After header expects integer seconds.

FIX:
  Rearrange parentheses so division happens first, then ceil:

    resetAfter: Math.ceil((windowMs - (now % windowMs)) / 1000),

  Now: Math.ceil(21962 / 1000) = Math.ceil(21.962) = 22  ← integer


BUG #8  —  Middleware: No Retry-After Header on 429 Response
=============================================================

FILE:  src/middleware/rateLimiter.js, lines 51-56

CURRENT CODE:
    if (!result.allowed) {
      return res.status(429).json({
        error:      "Too Many Requests",
        retryAfter: result.resetAfter,
        message:    `Rate limit exceeded. Try again in ${result.resetAfter}s`,
      });
    }

PROBLEM:
  The retryAfter value is only in the JSON body. Per RFC 6585 / RFC 7231,
  a 429 response SHOULD include a "Retry-After" HTTP header. Many HTTP
  clients and load balancers rely on this header to know when to retry.
  Without it, clients must parse the response body — fragile and non-
  standard.

FIX:
  Add the header in the res.set() call above, OR add it to the 429
  response:

    if (!result.allowed) {
      res.set("Retry-After", String(Math.ceil(result.resetAfter)));
      return res.status(429).json({
        error:      "Too Many Requests",
        retryAfter: result.resetAfter,
        message:    `Rate limit exceeded. Try again in ${result.resetAfter}s`,
      });
    }

  The String() cast ensures it's a valid header value. Math.ceil
  guarantees an integer as required by the spec.


BUG #10  —  Middleware: No Input Validation
============================================

FILE:  src/middleware/rateLimiter.js, lines 24-30

CURRENT CODE:
    const {
      algorithm    = "slidingWindowCounter",
      maxRequests  = parseInt(process.env.DEFAULT_MAX_REQUESTS) || 100,
      windowMs     = parseInt(process.env.DEFAULT_WINDOW_MS)    || 60000,
      keyGenerator = (req) => req.ip,
    } = options;

PROBLEM:
  No validation on maxRequests or windowMs. Calling:
    rateLimiter({ maxRequests: -5, windowMs: 0 })
  produces nonsensical behavior: negative limits, division by zero in
  refillRate calculation (tokenBucket), infinite loops, etc.

FIX:
  Add validation at the top of the rateLimiter function:

    function rateLimiter(options = {}) {
      const {
        algorithm    = "slidingWindowCounter",
        maxRequests  = parseInt(process.env.DEFAULT_MAX_REQUESTS) || 100,
        windowMs     = parseInt(process.env.DEFAULT_WINDOW_MS)    || 60000,
        keyGenerator = (req) => req.ip,
      } = options;

      if (maxRequests < 1) {
        throw new Error(`maxRequests must be >= 1, got ${maxRequests}`);
      }
      if (windowMs < 1000) {
        throw new Error(`windowMs must be >= 1000ms, got ${windowMs}`);
      }
      if (typeof keyGenerator !== "function") {
        throw new Error("keyGenerator must be a function");
      }

      // ... rest of function
    }
